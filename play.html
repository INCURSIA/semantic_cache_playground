Your cache can hallucinate.
Semantic caching doesn't compare strings, it compares meaning. Two queries that "mean the same thing" return the cached answer instantly. But what happens when two queries sound identical yet need completely different answers?
How semantic caching works

When a query arrives, the system generates a vector embedding, a list of numbers representing meaning. Instead of checking "is this the exact same string?", it asks "is this meaning close enough?" If the similarity score exceeds a threshold, the cached response is served. No LLM call. No cost. No latency.

The risk: two queries can score 0.96+ similarity while requiring completely different answers. One wrong digit. One wrong name. The cache doesn't know the difference.

The solution
Check facts before checking vectors.
A hybrid key adds a strict layer before vector comparison. Critical entities dosages, account IDs, drug names are extracted from the prompt and embedded directly in the Redis key.

The cache now has two gates: Gate 1 compares extracted entities exactly (string match). Gate 2 compares vector embeddings (similarity score).

If Gate 1 fails, Gate 2 never runs. The vectors are never compared. A $0.003 LLM call replaces a free but wrong cached answer.
